# 架构设计文档

## 1. 技术选型

### 核心技术栈
- **语言**: 原生 HTML + CSS + JavaScript (ES6+)
- **存储**: 浏览器 LocalStorage API
- **构建工具**: 无需构建工具，直接运行，降低复杂度
- **部署**: 可以直接静态部署到任何静态文件服务器

### 选型理由
1. 项目需求简单，原生JS足够满足需求，无需引入框架依赖
2. 零构建配置，打开浏览器即可运行，部署简单
3. 体积小，加载速度快，用户体验好
4. 对于笔记应用来说，LocalStorage完全满足存储需求

## 4. 模块划分

### 4.1 storage.js - 存储模块
**职责**: 封装LocalStorage操作，提供统一的数据存取接口

**暴露方法**:
- `loadNotes()`: 加载所有笔记
- `saveNotes(notes)`: 保存笔记数组到存储
- `clearNotes()`: 清空所有笔记

### 4.2 notes.js - 业务逻辑模块
**职责**: 处理笔记相关的核心业务逻辑

**暴露方法**:
- `createNote()`: 创建新笔记
- `getNotes()`: 获取所有笔记
- `getNoteById(id)`: 根据ID获取笔记
- `updateNote(id, content)`: 更新笔记内容
- `deleteNote(id)`: 删除笔记
- `extractTitle(content)`: 从内容提取标题

### 4.3 ui.js - UI交互模块
**职责**: 处理所有DOM操作和用户交互

**暴露方法**:
- `renderNoteList(notes, selectedId)`: 渲染笔记列表
- `renderEditor(content)`: 渲染编辑器内容
- `bindEvents(handlers)`: 绑定所有事件处理函数
- `showToast(message)`: 显示提示消息
- `showConfirmDialog(message, onConfirm, onCancel)`: 显示确认对话框
- `focusEditor()`: 聚焦编辑器

### 4.4 app.js - 入口模块
**职责**: 整合所有模块，初始化应用，处理整体流程

## 5. 核心流程设计

### 初始化流程
1. 页面加载完成 → 初始化应用
2. 从LocalStorage加载笔记 → 如果没有数据，初始化空数组
3. 渲染笔记列表 → 如果有笔记，默认选中第一个
4. 绑定所有用户交互事件 → 等待用户操作

### 创建笔记流程
1. 用户点击「新建笔记」按钮
2. 创建新笔记对象 → 添加到笔记数组
3. 保存到LocalStorage
4. 重新渲染笔记列表
5. 加载新笔记到编辑器 → 自动聚焦

### 选择笔记流程
1. 用户点击笔记列表项
2. 获取对应笔记对象
3. 加载笔记内容到编辑器
4. 重新渲染列表，标记当前选中项

### 保存笔记流程
1. 用户点击「保存」按钮
2. 获取当前选中笔记ID → 如果没有选中，提示用户
3. 获取编辑器内容 → 检查内容大小（LocalStorage一般限制5MB）
4. 更新笔记对象 → 更新时间戳
5. 保存到LocalStorage → 重新渲染列表
6. 显示保存成功提示

### 删除笔记流程
1. 用户点击「删除」按钮
2. 检查是否有选中笔记 → 没有则提示
3. 弹出二次确认对话框
4. 用户确认 → 从数组中删除笔记 → 保存到LocalStorage
5. 清空编辑器 → 重新渲染列表 → 提示删除成功
6. 用户取消 → 关闭对话框，不做任何操作

## 6. 错误处理

### 可能的错误场景
1. **LocalStorage不可用**: 浏览器隐私模式可能禁用LocalStorage → 提示用户
2. **存储容量超出**: 内容过大超出LocalStorage容量限制 → 提示用户清理旧笔记
3. **数据损坏**: 存储的JSON数据解析失败 → 清空损坏数据，初始化应用
4. **选中笔记不存在**: 数据不一致 → 清空编辑器，提示用户

---

## 7. 性能优化

1. 最小DOM操作：只有数据变化时才重新渲染，不做全量重复渲染
2. 事件委托：使用事件委托减少事件监听器数量
3. 惰性操作：不需要立即计算的内容延迟计算
4. CSS使用flexbox布局，性能优秀，适配性好

---

## 架构总结

本架构设计遵循以下原则：
1. **简单清晰**: 模块化拆分清晰，每个模块职责单一，易于理解和维护
2. **无依赖**: 不引入第三方框架，原生JS实现，减少复杂度
3. **可扩展**: 模块化设计便于后续功能扩展
4. **易部署**: 纯静态文件，无需后端，部署极其简单

完全满足当前项目需求，同时保持了足够的灵活性。
