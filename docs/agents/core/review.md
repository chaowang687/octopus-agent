## 7. 工作流程

### 流程图

```
测试完成 → 代码审查 → 安全分析 → 质量评估 → 审查报告
    ↓          ↓          ↓          ↓          ↓
  读取代码   检查质量   检查安全   评估质量   生成报告
```

### 详细步骤

1. **代码审查**
   - 读取代码文件
   - 检查代码质量
   - 检查代码规范
   - 识别潜在问题

2. **安全分析**
   - 检查安全漏洞
   - 检查敏感信息
   - 检查权限问题
   - 评估安全风险

3. **质量评估**
   - 评估代码质量
   - 评估可维护性
   - 评估可扩展性
   - 评估性能

4. **审查报告**
   - 生成审查报告
   - 提供改进建议
   - 决定能否交付
   - 提交审查结果

## 8. 输入输出规范

### 输入格式

```typescript
interface ReviewInput {
  projectPath?: string;       // 项目路径
  testResult?: TestResult;    // 测试结果
  context?: Record<string, any>; // 上下文信息
}
```

### 输出格式

```typescript
interface ReviewOutput {
  passed: boolean;             // 是否通过
  issues: string[];            // 问题列表
  suggestions: string[];       // 改进建议
  summary: string;             // 审查总结
}
```

## 9. 错误处理

### 常见错误

| 错误类型 | 错误信息 | 处理方式 |
|---------|---------|---------|
| ReviewError | 审查失败 | 检查代码文件 |
| SecurityIssue | 安全问题 | 标记安全问题 |

### 错误恢复策略

1. **审查失败**
   - 检查代码文件
   - 检查文件权限
   - 重新审查

2. **安全问题**
   - 标记安全问题
   - 提供修复建议
   - 要求修复后重新审查

### C. FAQ

**Q1: 代码审查员智能体能修改代码吗？**
A: 不能，代码审查员智能体只负责审查和报告问题，不修改代码。

**Q2: 代码审查员智能体如何评估代码质量？**
A: 检查代码规范、可维护性、可扩展性、安全性等方面，综合评估代码质量。

**Q3: 代码审查员智能体的输出为什么这么简单？**
A: 为了让用户能理解审查结果，使用日常语言描述问题，避免使用技术术语。
