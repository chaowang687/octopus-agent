- 提供清晰的问题描述

### 不包含的功能
- 不修复代码问题
- 不进行代码实现
- 不修改测试代码

## 6. System Prompt

```text
你是测试工程师。你需要：

1. 试着用用户的方式操作这个应用
2. 看看有没有什么问题
3. 用简单的语言告诉用户发现的问题

**【重要】输出要求 - 必须遵守！**

禁止输出：
- ❌ 技术术语
- ❌ 堆栈跟踪信息
- ❌ 复杂的错误代码

只输出：
- ✅ "我发现了一个问题：点击按钮后页面卡住了"
- ✅ "功能正常，我测试了几种情况都能工作"
- ✅ "有一个小问题：文字显示不全"

**输出格式**：
{
  "passed": true/false,
  "issues": ["用简单语言描述的问题1", "问题2"],
  "severity": "critical/major/minor"
}

如果有问题，明确告诉用户"我测试时发现了一些问题，需要修复"。如果没有问题，告诉用户"测试通过了！"
```

## 7. 工作流程

### 流程图

```
开发完成 → 测试设计 → 测试执行 → 问题报告 → 回归测试
    ↓          ↓          ↓          ↓          ↓
  理解功能   设计用例   执行测试   报告问题   验证修复
```

### 详细步骤

1. **测试设计**
   - 理解功能需求
   - 设计测试用例
   - 确定测试场景
   - 准备测试数据

2. **测试执行**
   - 执行功能测试
   - 执行边界测试
   - 执行异常测试
   - 记录测试结果

3. **问题报告**
   - 识别问题
   - 评估问题严重程度
   - 编写问题报告
   - 提交问题报告

4. **回归测试**
   - 验证问题修复
   - 执行回归测试
   - 确保无新问题
   - 生成测试报告

## 8. 输入输出规范

### 输入格式

```typescript
interface TestInput {
  features: string[];         // 功能列表
  projectPath?: string;       // 项目路径
  context?: Record<string, any>; // 上下文信息
}
```

### 输出格式

```typescript
interface TestOutput {
  passed: boolean;             // 是否通过
  issues: string[];            // 问题列表
  severity: 'critical' | 'major' | 'minor'; // 严重程度
  summary: string;             // 测试总结
}
```

### 示例

**输入示例**：
```json
{
  "features": [
    "添加新的待办事项",
    "删除不需要的事项",
    "标记事项为完成状态"
  ]
}
```

**输出示例**：
```json
{
  "passed": false,
  "issues": [
    "点击添加按钮后，待办事项没有显示",
    "删除按钮点击后，事项没有被删除"
  ],
  "severity": "major",
  "summary": "测试发现2个主要问题，需要修复"
}
```

## 9. 错误处理

### 常见错误

| 错误类型 | 错误信息 | 处理方式 |
|---------|---------|---------|
| TestError | 测试执行失败 | 检查测试环境 |
| TimeoutError | 测试超时 | 检查应用状态 |
| EnvironmentError | 环境错误 | 检查配置 |

### 错误恢复策略

1. **测试执行失败**
   - 检查测试环境
   - 检查应用状态
   - 重新执行测试

2. **测试超时**
   - 检查应用响应
   - 增加超时时间
   - 重新执行测试

3. **环境错误**
   - 检查配置
   - 重启应用
   - 重新执行测试

### C. FAQ

**Q1: 测试工程师智能体能修复代码问题吗？**
A: 不能，测试工程师智能体只负责测试和报告问题，不修复代码。

**Q2: 测试工程师智能体如何评估问题严重程度？**
A: 根据问题对用户体验和功能的影响程度评估，分为critical、major、minor三个级别。

**Q3: 测试工程师智能体的输出为什么这么简单？**
A: 为了让用户能理解测试结果，使用日常语言描述问题，避免使用技术术语。
